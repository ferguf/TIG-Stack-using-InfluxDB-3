# 💰 LRIC Cost Model Database Design

## 1. Overview

This document describes the design of the table used to establish the **Long Run Incremental Cost (LRIC) model parameters**. This data is separated from the core hardware specs to allow for complex cost scenarios, versioning, and modeling based on usage/fill levels, which are independent of the hardware's static physical properties.

## 2. Table Schema

### 2.1. `lric_cost_model` (Cost Calculation Parameters Table)

Stores the base costs and utilization factors necessary to derive the final LRIC cost for a specific asset configuration.

| Column Name | Data Type | Description | Constraints / Notes |
| :--- | :--- | :--- | :--- |
| `cost_model_id` | `UUID` | **Primary Key.** Unique ID for this cost scenario/model. | PK, `gen_random_uuid()` |
| **`model_id`** | `UUID` | **Foreign Key** to the entity being costed (e.g., `hardware_specs.hardware_id`). | NOT NULL |
| `model_type` | `VARCHAR(50)` | Defines the type of entity referenced (e.g., 'Hardware', 'Role', 'Service'). | NOT NULL |
| `description` | `VARCHAR(255)` | Descriptive title for the cost model (e.g., "100G Port Cost with 25% Fill"). | |
| **Cost Metrics** | | | |
| `base_cost` | `DECIMAL(15, 4)` | The calculated cost (\$) before applying fill/utilization factors. | NOT NULL |
| `min_fill` | `INTEGER` | Minimum required utilization (0-100). | |
| `lric_fill` | `INTEGER` | **Target utilization (0-100)** used in the LRIC formula. | |
| **Capacity/Level Metrics** | | | |
| `type` | `VARCHAR(50)` | The cost dimension (e.g., '100G', '400G', '$/G'). | |
| `level` | `DECIMAL(15, 4)` | The specific capacity or quantity the cost applies to (e.g., 100 for 100G). | |
| `min_level` | `DECIMAL(15, 4)` | A minimum quantity threshold. | |
| **Calculated Output** | | | |
| `calculated_lric_cost` | `DECIMAL(15, 4)` | **Generated Column** calculated as: $base\_cost \times (\frac{lric\_fill}{100})$. | GENERATED ALWAYS |
| **Auditing** | | | |
| `created_at` | `TIMESTAMPTZ` | Timestamp of model creation. | Auto-set |
| `updated_at` | `TIMESTAMPTZ` | Timestamp of last modification. | Triggered |

## 3. Calculation Formula

The calculated LRIC cost is automatically generated by PostgreSQL based on the two core inputs:

$$
\text{calculated\_lric\_cost} = \text{base\_cost} \times \frac{\text{lric\_fill}}{100}
$$

This ensures that the stored calculated value is always consistent with the currently defined `base_cost` and `lric_fill` percentages.

## 4. Key Relationships

This table serves as a model definition. It is linked to the physical asset by joining the `lric_cost_model.model_id` to `hardware_specs.hardware_id`.

**Example Join Path:**
To find the LRIC cost model for a specific device:

`devices` $\rightarrow$ `hardware_specs` $\rightarrow$ `lric_cost_model`


## 5. Technology and Implementation

* **Database:** PostgreSQL
* **Key Feature:** Uses a `GENERATED ALWAYS` column to guarantee the integrity of the calculated cost field.
* **Auditing:** Utilizes the same `set_updated_at_timestamp` trigger as the hardware tables for consistency.
