# Design Guide: Fabric Connection Management

## 1. Module Overview

The **Fabric Connection** module acts as the logical "patch cable" of the software-defined network. It defines the point-to-point or multipoint relationship between ports within a specific `FabricService`. While a `FabricService` represents a customer's overall product, the `FabricConnection` represents the actual configuration of VRFs, VLANs, and bandwidth.

---

## 2. Technical Architecture

### Data Relationship Model

This module implements a complex many-to-one relationship with a "Self-Referencing" flavor on the `Port` table:

* **Service Anchor**: Every connection must belong to a parent `service_id`.
* **Dual-Port Anchoring**: Each record tracks two distinct endpoints (`port_a_id` and `port_b_id`).
* **Foreign Key Disambiguation**: Because both ports reference the same `ports` table, the model explicitly defines `foreign_keys` to prevent SQLAlchemy join ambiguity.

```python
# Disambiguating multiple links to the same table
port_a = relationship("Port", foreign_keys=[port_a_id])
port_b = relationship("Port", foreign_keys=[port_b_id])

```

## 3. Key Attributes & Logic

### Network Configuration Metadata

| Attribute | Purpose | Logic |
| --- | --- | --- |
| `vrf_name` | Virtual Routing | Segregates traffic at Layer 3. |
| `s_vlan` | Service VLAN | The outer "provider" tag. |
| `c_vlan_list` | Customer VLANs | A string-encoded list of inner tags allowed on the circuit. |
| `service_bw` | Bandwidth | Integer representing Mbps/Gbps for rate-limiting. |

### The "Partial Update" Pattern

To support modern API interactions, the `update_fabric_connection` logic utilizes `exclude_unset=True`. This allows the frontend to send only the fields that changed (e.g., just updating the `health_status`) without needing to provide the entire object, preventing accidental data overwrites.

---

## 4. Operation Lifecycle

### Creation Sequence

1. **Schema Injection**: `FabricConnectionIn` validates UUIDs and integer ranges.
2. **Persistence**: `api_operation.post_fabric_connection` initializes the ORM object and assigns a fresh `uuid4()`.
3. **Traceability**: The `created_at` and `updated_at` timestamps are generated by the database engine (server-side) to ensure clock synchronization across the cluster.
4. **Validation**: `model_validate` converts the DB-resident object back into a Pydantic model for the HTTP response.

---

## 5. Implementation Notes

> [!TIP]
> **Debugging Protocol**: This module includes extensive `DEBUG` print statements at the `api_operation` level. In a production environment, these should be transitioned to a structured logging framework (like `structlog` or `logging.getLogger`) to monitor state changes without cluttering standard output.

### Future Improvements

* **VLAN Validation**: Implement a validator to ensure `s_vlan` is within the 1-4094 range.
* **Port Availability Check**: Add a check during `POST` to ensure `port_a` and `port_b` are not already occupied by another active fabric connection.
